# Custom Commands — Design

**Date:** 2026-04-29
**Status:** Approved
**Scope:** Add user-configurable custom commands to Settings; expose them in the Files page right-click menu; execute `<command> [static args...] <selected-path>` for each selected file.

---

## 1. Goal

Allow the user to configure named shell commands in Settings (e.g. `bup 上传` → `bup`), and trigger them on selected files from the Files page right-click menu. Each invocation runs `<command> <selected-path>` as a separate child process.

Concrete example: user adds `(label="bup 上传", command="bup")`. Right-clicking a file `A` in Files shows a "bup 上传" menu item; clicking it runs `bup A` from the file's parent directory.

## 2. Data Model

### 2.1 `CustomCommand`

```ts
// src/types.ts
export interface CustomCommand {
  id: string          // uuid v4, generated on add
  label: string       // shown in menu and status bar; non-empty
  command: string     // shlex-parsed; first token is the binary, rest are static args
}
```

### 2.2 Settings extension

```ts
// src/types.ts — extend AppSettings
```

## 3. Execution & IPC

### 3.1 New service: `electron/services/customCommandRunner.ts`

```ts
import { spawn } from 'node:child_process'
import * as path from 'node:path'
import { getShellPath } from './shellPath'
import { parseShellArgs } from './shlex'  // pure function, see below

export interface RunRequest {
  runId: string       // generated by the IPC handler before invoking
  commandId: string
  paths: string[]
}

export type ProgressEvent =
  | { type: 'start'; runId: string; commandLabel: string; total: number }
  | { type: 'advance'; runId: string; done: number; current: string }
  | { type: 'fileError'; runId: string; path: string; message: string; stderr: string }
  | { type: 'finish'; runId: string; ok: number; failed: number }

export class CustomCommandRunner {
  // sequential queue per runId; one runId at a time is fine for v1
  async run(req: RunRequest, settings: AppSettings, emit: (e: ProgressEvent) => void): Promise<void>
}
```

Behavior:

- Look up the `CustomCommand` by `commandId`. If missing, emit `finish` with all-failed and return.
- `parseShellArgs(cmd.command)` → `[bin, ...staticArgs]`. If parsing fails or `bin` is empty, emit one `fileError` per path with a parse-error message and a `finish`.
- Emit `start { total: paths.length }`.
- For each path **sequentially** (`for...of` with `await`):
  - `cwd = path.dirname(filePath)`
  - `argv = [...staticArgs, filePath]`
  - `spawn(bin, argv, { cwd, shell: false, env: { ...process.env, PATH: getShellPath() } })`
  - Collect stderr (last 4KB only, drop overflow). stdout is fully discarded.
  - Wait for `close` event. code===0 → success; otherwise emit `fileError { path, message: 'exit code N', stderr }`.
  - On `error` event (ENOENT etc.) → emit `fileError { path, message: 'notFound:<bin>', stderr: '' }` and continue with next path.
  - After each path, emit `advance { done: i+1, current: basename }`.
- After loop, emit `finish { ok, failed }`.

`shell: false` is non-negotiable — the path is appended as a separate argv entry, so spaces and special characters cannot be interpreted as shell metacharacters.

### 3.2 New helper: `electron/services/shlex.ts`

Pure function `parseShellArgs(input: string): string[]` that handles:

- Whitespace splitting
- Single quotes (no escapes inside)
- Double quotes (with `\"`, `\\` escapes)
- Backslash escape outside quotes
- Throws `Error('unclosed quote')` if quotes are unbalanced

We implement this ourselves (~30 lines) rather than pulling in the npm `shell-quote` dependency. Unit-tested in isolation.

### 3.3 New helper: `electron/services/shellPath.ts`

macOS GUI apps inherit Spotlight/launchd's PATH, which often excludes `/usr/local/bin`, `~/.local/bin`, Homebrew paths, etc. We mimic the VS Code pattern:

```ts
let cached: string | null = null
let inFlight: Promise<string> | null = null

export async function initShellPath(): Promise<void> {
  if (cached) return
  inFlight ??= resolveShellPath()
  cached = await inFlight
}

export function getShellPath(): string {
  return cached ?? process.env.PATH ?? ''
}

async function resolveShellPath(): Promise<string> {
  // spawn('bash', ['-lc', 'echo $PATH']) with a 2s timeout
  // on success: return stdout.trim()
  // on failure or timeout: return process.env.PATH ?? ''
}
```

Called from `electron/main.ts` `app.whenReady()` (fire-and-forget — don't block IPC registration). `getShellPath()` is synchronous so the runner doesn't await on every invocation.

### 3.4 IPC contract

Update `src/types.ts`:

```ts
export interface DashMacAPI {
  // ...existing
  runCommand(commandId: string, paths: string[]): Promise<{ ok: true; runId: string } | { ok: false; message: string }>
  onCommandProgress(callback: (event: ProgressEvent) => void): () => void
}
```

Wire in `electron/preload.ts`:
- `runCommand` → `ipcRenderer.invoke('query:run-command', commandId, paths)`
- `onCommandProgress` → `ipcRenderer.on('cmd:progress', ...)` returning unsubscribe

Wire in `electron/main.ts:registerIpcHandlers`:
- `ipcMain.handle('query:run-command', (_e, commandId, paths) => { const runId = randomUUID(); runner.run({ runId, commandId, paths }, settings, emit).catch(logErr); return { ok: true, runId } })`
- The handler returns `{ ok: true, runId }` immediately; the runner continues in the background, emitting events through `cmd:progress`.
- `emit` sends to all open windows: `mainWindow?.webContents.send('cmd:progress', e)` and same for `trayWindow`.

### 3.5 Concurrency model

Each `run()` call manages its own sequential pipeline (paths run one at a time within a run). Multiple `run()` calls — e.g., starting a second command while the first is still going — execute concurrently as independent pipelines. Each has its own `runId` and its own row in the status strip; the strip stacks rows vertically.

### 3.6 No cancel in v1

Once a run starts, it runs to completion. The status bar shows progress so the user knows where it is. Cancel is deferred to v2.

## 4. Right-Click Menu Integration

### 4.1 Menu position

Modify `src/components/files/FilesPage.tsx` `rowMenuItems(entry)` (around line 222). After existing items, conditionally append:

```ts
const customCmds = useSettingsStore(s => s.settings.customCommands)
// ...
const baseItems = [/* existing items */]
if (customCmds.length === 0) return baseItems
return [
  ...baseItems,
  { type: 'separator' },
  ...customCmds.map(c => ({
    label: c.label,
    onClick: () => handleRunCommand(c.id, entry),
  })),
]
```

If `customCommands.length === 0`, no separator and no command items appear.

### 4.2 Multi-select rule (Finder-style)

```ts
function handleRunCommand(commandId: string, clickedEntry: FileEntry) {
  const targetPaths = selectedPaths.has(clickedEntry.path)
    ? Array.from(selectedPaths)        // run on all selected
    : [clickedEntry.path]              // run on just the right-clicked one
  window.api.runCommand(commandId, targetPaths)
}
```

Both files and directories are eligible — no enabled/disabled state based on `entry.isDirectory`.

### 4.3 Status bar + toast UI

New Zustand store `src/stores/commandRunStore.ts`:

```ts
interface Run {
  runId: string
  commandLabel: string
  total: number
  done: number
  current?: string
  failures: { path: string; message: string; stderr: string }[]
}

interface CommandRunStore {
  runs: Run[]                          // active only; cleared on finish after toast
  start: (e: StartEvent) => void
  advance: (e: AdvanceEvent) => void
  fileError: (e: FileErrorEvent) => void
  finish: (e: FinishEvent) => void
}
```

New component `src/components/common/CommandRunStatus.tsx` mounted at App root next to `ToastRoot`:

- Subscribes to `commandRunStore.runs`.
- Renders a fixed bottom-right narrow strip when `runs.length > 0`. Multiple concurrent runs stack vertically (newest on top):
  ```
  bup 上传 · 2/5 · README.md
  git status · 1/3 · src/index.ts
  ```
- On `finish`:
  - All success → `toast.success("bup 上传完成 (5)")`
  - Some failed → `toast.warning("bup 上传完成 5 项，2 项失败", { action: { label: '查看详情', onClick: openDetailsDialog(failures) } })`
  - All failed → same shape with `commandRun.allFailed`
- Failure-details dialog: a simple `<dialog>` listing each `{ path, message, stderr (truncated to 1KB) }`. Plain text, monospace, copyable.

App root mounts the IPC subscription once:

```ts
// src/App.tsx (MainApp) — single useEffect
useEffect(() => {
  return window.api.onCommandProgress((e) => {
    const store = useCommandRunStore.getState()
    switch (e.type) {
      case 'start':     store.start(e); break
      case 'advance':   store.advance(e); break
      case 'fileError': store.fileError(e); break
      case 'finish':    store.finish(e); break
    }
  })
}, [])
```

The tray window does NOT mount this subscription (tray is read-only system metrics).

## 5. Testing Strategy

### 5.1 Unit tests (Vitest, `tests/electron/`)

**`tests/electron/shlex.test.ts`** (new):
- `parseShellArgs('bup')` → `['bup']`
- `parseShellArgs('bup --foo "a b"')` → `['bup', '--foo', 'a b']`
- `parseShellArgs("bup 'a b'")` → `['bup', 'a b']`
- `parseShellArgs('bup "a\\"b"')` → `['bup', 'a"b']`
- `parseShellArgs('bup "unclosed')` → throws `'unclosed quote'`
- `parseShellArgs('   ')` → `[]`

**`tests/electron/customCommandRunner.test.ts`** (new):
- Mock `child_process.spawn` with an EventEmitter that exposes a `stdout`/`stderr` Readable and emits `close` / `error` on demand.
- Success path: 1 path, code=0 → events: start → advance(1/1) → finish(ok=1, failed=0).
- Failure path: 1 path, code=1, stderr="boom" → events: start → fileError(message='exit code 1', stderr='boom') → advance → finish(ok=0, failed=1).
- ENOENT: spawn emits 'error' with `code='ENOENT'` → fileError with `notFound:<bin>` → advance → finish.
- Sequential: 3 paths, assert second `spawn` is not called until first emits `close`.
- Unknown commandId → finish with all-failed and a parse-error message.
- Empty `command` field → same as parse error.
- argv shape: spawn called with `(bin, [...staticArgs, filePath], { cwd: dirname, shell: false, env: containing PATH })`.
- stderr capped at 4KB (write 8KB, assert recorded length ≤ 4096).

**`tests/electron/shellPath.test.ts`** (new):
- Mock `spawn('bash', ['-lc', 'echo $PATH'])`. `getShellPath()` returns the stdout string.
- Spawn fails (emits 'error') → falls back to `process.env.PATH`. No throw.
- Timeout (no close within 2s using `vi.useFakeTimers()`) → falls back. No throw.
- Cache: second `initShellPath()` call does not invoke spawn again.

**`tests/electron/settingsStore.test.ts`** (extend):
- DEFAULTS contain `customCommands: []`.
- Loading a settings.json without `customCommands` field merges in `[]`.

**`tests/i18n.test.ts`** (existing): automatically validates the new keys exist in both locales with identical shape.

### 5.2 Manual QA checklist

- Configure `bup 上传` / `bup`; right-click a file → menu shows item → click → status strip appears → completion toast.
- Configure `notexist` / `notexist`; right-click → toast says "找不到命令 notexist".
- Multi-select 5 files; right-click one of the selected → executes 5 times sequentially, status strip counts 1/5 → 5/5.
- Select a directory + a file, multi-run; both succeed.
- File path with spaces (`~/My Docs/a.txt`) → command receives full path (verifiable with `printf "%s\n"` as the configured command).
- Launch DashMac.app from Finder (not from terminal); configure a binary in `~/.local/bin` → command runs successfully (validates `shellPath`).
- Empty command field in Settings → "保存" disabled.
- Unclosed quote in command → red error message; "保存" disabled.
- Delete a command in Settings → row disappears; right-click menu no longer offers it.
- `customCommands: []` (default) → right-click menu has no separator and no extra items.

### 5.3 Out of test scope

- Renderer React component tests (project has no precedent; `vitest.config.ts` only includes `tests/`).
- End-to-end spawn of real binaries (mocking covers all branches; CI may not have a stable shell environment).

## 6. Out of Scope (v1)

- Cancelling a running command.
- Variable substitution other than the trailing path append (no `{name}`, `{ext}`, `{dirname}` placeholders).
- Per-command environment variable editing.
- Command grouping, icons, keyboard shortcuts, ordering UI (commands appear in their array order; reorder lands in v2 if requested).
- Windows / Linux compatibility (DashMac is macOS-only).
- Persisted run history.
- Reordering / grouping commands in Settings UI (commands are displayed in their stored array order; reorder UI deferred to v2 if requested).

## 7. Risks & Mitigations

1. **Login-shell PATH resolution slow or fails** → command not found.
   *Mitigation:* `initShellPath()` runs at `app.whenReady()` with a 2s timeout, doesn't block IPC. Fallback to `process.env.PATH`. The `notFound` error message explicitly mentions PATH.

2. **Long-running command blocks queue** → user picks 10 files × 30s command = 5 minutes apparent freeze.
   *Mitigation:* Status strip stays visible with current progress. Cancel deferred to v2 with a clear story rather than rushed.

3. **Large stdout/stderr** → memory bloat (`find /` outputs gigabytes).
   *Mitigation:* stdout fully discarded (`subprocess.stdout.resume()`); stderr capped at last 4KB (rolling buffer).

4. **Command injection** → user-supplied command could be malicious.
   *Out of scope as risk:* the user is configuring commands for themselves on their own machine. `shell: false` ensures we don't add our own injection surface — file paths can never be interpreted as flags or metacharacters.

5. **Settings.json write failure** → existing `settingsStore` already wraps writes in try/catch and logs; no new failure mode.

6. **`customCommandRunner` running while window closes** → orphan process.
   *Mitigation:* Track active `ChildProcess` references; on `before-quit`, send `SIGTERM` to all and wait up to 1s before `SIGKILL` and exit. (Existing close handlers already coordinate scheduler shutdown — this slots in.)
