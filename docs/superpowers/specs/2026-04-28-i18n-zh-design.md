# Multi-Language Support (i18n) — Simplified Chinese

**Date:** 2026-04-28
**Status:** Design approved, pending implementation plan

## Goals

Add Simplified Chinese (`zh-CN`) localization to DashMac alongside the existing English UI. The user picks a language preference of `auto` (follow system locale), `English`, or `简体中文`, and the choice applies immediately across all surfaces — main window, tray panel, and the macOS tray context menu — without restarting the app.

## Non-Goals

- Byte-unit localization (`B`/`KB`/`MB`/`GB` stay English; matches macOS Activity Monitor convention).
- Locale-aware number / date formatting via `Intl`.
- Traditional Chinese, Japanese, or any third language. Architecture allows adding them later by dropping a new locale file in.
- README / documentation translation. `README.ZH.md` already exists and is independent of in-app strings.
- Lazy / dynamic locale loading. Total dictionary size is under ~10 KB; static `import` is simpler.
- Right-to-left layouts.
- Translation management platform (Crowdin, Lokalise). Strings are hand-maintained TS files.

## Architecture

### Two parallel translation tables

The Electron main process cannot import renderer code, so each side ships its own dictionary:

- `src/i18n/` — renderer (~60–80 strings: sidebar, header, settings, dashboard cards, tray panel)
- `electron/i18n/` — main process (~3–5 strings: tray tooltip, tray context menu items)

Both tables share the same key-naming style and the same `resolveLang` algorithm.

### Single source of truth: `AppSettings.language`

The user's preference lives in `AppSettings.language: 'auto' | 'en' | 'zh-CN'`, persisted by the main process to `app.getPath('userData')/settings.json`. The main process also computes a derived read-only field `resolvedLanguage: 'en' | 'zh-CN'` and returns it in `getSettings()`. The renderer never writes `resolvedLanguage`; main ignores it on `saveSettings`.

### Startup flow

1. Main reads `settings.json` (missing or corrupt → defaults, including `language: 'auto'`).
2. Main runs `resolveLang(stored.language, app.getLocale())` → resolved `'en' | 'zh-CN'`.
3. Main sets module-level `currentLang` and builds the tray with translated labels.
4. Renderer's `main.tsx` does an async bootstrap: `await window.api.getSettings()`, then mounts `<I18nProvider initialLang={resolvedLanguage}>` around `<App />`. This avoids a flash of English on first paint.

### Language change flow

1. User opens Settings → changes the language `<select>`.
2. Renderer calls `window.api.saveSettings({...settings, language: next})`.
3. Main writes `settings.json`, recomputes resolved lang, calls `setLang(newResolved)`, and rebuilds the tray context menu via `rebuildTrayMenu()`.
4. Main broadcasts `'i18n:lang-changed'` with the new resolved lang to all open `BrowserWindow`s (main window and tray window) via `sendToAllWindows`.
5. Each window's `<I18nProvider>` listens via `window.api.onLangChanged(...)` and updates its Context state, triggering a re-render of the entire app.

This gives a single unidirectional data flow: the renderer never optimistically updates its own state — it always waits for the main-process broadcast, so the three surfaces (main window, tray window, tray menu) can never drift.

### Language resolution

```ts
function resolveLang(pref: 'auto' | 'en' | 'zh-CN', systemLocale: string): 'en' | 'zh-CN' {
  if (pref !== 'auto') return pref
  return systemLocale.toLowerCase().startsWith('zh') ? 'zh-CN' : 'en'
}
```

Any system locale starting with `zh` (`zh-CN`, `zh-Hans`, `zh-TW`, `zh-HK`, …) maps to `zh-CN`. This is intentional: the project supports two language buckets — Chinese vs everything else — and traditional-Chinese users will see Simplified Chinese, which is acceptable for v1.

The main-process dictionary is much smaller and covers only the tray surfaces:

```ts

```

Renderer `tray.*` (used by the popover panel) and main `tray.*` (used by tooltip + native context menu) live in separate files; the shared prefix is conceptual, not technical.

### Self-language names

The language `<select>` always renders option labels using the **target language's name in itself**:
- "English" stays `English` in both dictionaries
- 简体中文 stays `简体中文` in both dictionaries

So users always recognize the language they want to switch to, regardless of the current display language.

### Strings that stay literal (not in dictionaries)

- Byte units: `B`, `KB`, `MB`, `GB`, `TB`
- Version string `v0.1.0`
- App name `DashMac` (the dictionary entries `tray.tooltip` (main) and `tray.name` (renderer) are both `'DashMac'` in both languages — keys exist for future flexibility)
- IP addresses, interface names, percentages, formatted byte values
- Log lines and error messages (developer-facing, not surfaced in UI)

## Renderer API

```ts
// src/i18n/index.ts
export type Lang = 'en' | 'zh-CN'

export function I18nProvider(props: { initialLang: Lang; children: React.ReactNode }): JSX.Element

export function useTranslation(): {
  t: (key: string, vars?: Record<string, string | number>) => string
  lang: Lang
  setLang: (next: 'auto' | Lang) => Promise<void>
}
```

State is held in `useState` inside the Provider — Context is right for UI-tree-scoped state. We do not use Zustand here.

`setLang` does **not** call `setState` directly. It calls `window.api.saveSettings({...current, language: next})` and waits for the main-process broadcast on `'i18n:lang-changed'` to flow back through `useEffect` and trigger the actual state change. Single unidirectional flow; no client-side optimistic update; impossible for renderer state to diverge from main.

`t(key, vars?)` implementation:
1. Look up `key` (dot-path) in the active language dictionary.
2. Miss → fall back to English dictionary.
3. Miss in both → return the key string itself; emit `console.warn` only when `import.meta.env.DEV`.
4. If `vars` provided, replace `{name}` tokens in the result with `String(vars.name)`.

Lookup + interpolation are extracted to `src/i18n/lookup.ts` as pure functions for testability.

### App integration

```tsx
// src/main.tsx
const settings = await window.api.getSettings()
ReactDOM.createRoot(document.getElementById('root')!).render(
  <I18nProvider initialLang={settings.resolvedLanguage}>
    <App />
  </I18nProvider>
)
```

The Settings page's language `<select>` is **not** gated by the existing "Save Settings" button. Changing it persists immediately and the UI updates instantly. All other settings keep the existing deferred-save UX. This asymmetry is intentional: language is a display affordance — users expect it to take effect the moment they pick it.

## Main process API

```ts
// electron/i18n/index.ts
export type Lang = 'en' | 'zh-CN'
export type LangPref = 'auto' | Lang

let currentLang: Lang = 'en'

export function setLang(lang: Lang): void
export function getLang(): Lang
export function resolveLang(pref: LangPref, systemLocale: string): Lang
export function t(key: string): string  // no interpolation needed in main; KISS
```

### main.ts changes

1. **Startup:** load settings, resolve lang, set `currentLang` before `createTray()`.
2. **`createTray()`:** replace literal `'DashMac'` / `'Open DashMac'` / `'Quit'` with `t('tray.tooltip')` / `t('tray.open')` / `t('tray.quit')`.
3. **Extract `rebuildTrayMenu()`:** the current `createTray` does both. After the split, language-change paths only rebuild the menu, not the tray itself.
4. **`save:settings` handler:** stop being a stub. Persist to disk, recompute resolved lang, and on change call `setLang` + `rebuildTrayMenu()` + `sendToAllWindows('i18n:lang-changed', newResolved)`.
5. **`get:settings` handler:** stop being a stub. Return persisted settings plus computed `resolvedLanguage`.

### `electron/services/settingsStore.ts` (new)

```ts
const DEFAULTS: AppSettings = {
  realtimeInterval: 2000, historyInterval: 60000, retentionDays: 90,
  trayDisplayMetric: 'memory', launchAtLogin: false, language: 'auto',
  // resolvedLanguage is computed, not stored
}

export function loadSettings(filepath?: string): AppSettings
export function saveSettings(s: AppSettings, filepath?: string): void
```

- File is `app.getPath('userData')/settings.json` by default; `filepath` parameter exists for test injection.
- `loadSettings` returns `DEFAULTS` if the file is missing or JSON-corrupt (no throw).
- `saveSettings` strips `resolvedLanguage` before writing (it's a derived field).
- Synchronous `fs.writeFileSync` — small file, low write frequency, simpler than async.

### Translation style guidelines

- No trailing punctuation on labels and short phrases (Chinese style).
- Localize technical terms with industry conventions ("Memory" → "内存"; "Swap" → "交换"; "Cached" → "已缓存").
- Buttons use action verbs ("Save Settings" → "保存设置"; "Export CSV" → "导出 CSV").
- Keep length-constrained strings (sidebar nav, tray panel rows) short.
- Mixed CJK + Latin uses no extra space around units (`8.5 GB`, not `8.5  GB`).

## Testing

Tests follow the existing `tests/electron/` convention (main-process modules only; no renderer test infrastructure). New tests:

### `tests/electron/i18n.test.ts`

1. `resolveLang('auto', 'en-US')` → `'en'`
2. `resolveLang('auto', 'zh-CN')` → `'zh-CN'`

11. `saveSettings(s, tmpPath)` then `loadSettings(tmpPath)` → returns `s` (round-trip), with `resolvedLanguage` not persisted

### Out of scope for tests

- React Provider re-render behavior — no existing renderer test infra.
- Preload IPC bridge.
- Real Tray rendering (requires Electron runtime).
- Linguistic correctness of translations — handled by manual review.

## Risks and mitigations

1. **Flash of English on first paint** → async bootstrap in `main.tsx` (`await getSettings()` before `createRoot().render()`). Adds ~10 ms; acceptable.

2. **Surfaces drift out of sync** → main process is the single authority; all three surfaces re-render only after main broadcasts `'i18n:lang-changed'`.

3. **Concurrent writes on `settings.json`** → small file, low write frequency, synchronous `writeFileSync`. No race possible.

4. **`as const` strictness blocking `zh-CN.ts`** → `en.ts` is `as const`; `zhCN.ts` is typed as a structurally-equivalent `Record<string, string | NestedRecord>` so values may differ. Runtime shape test is the safety net.

5. **"Save Settings" button confusing UX for language** → language `<select>` persists on change, bypassing the Save button. Documented as intentional asymmetry.

6. **Dev-time visibility of missing keys** → `t()` emits `console.warn` only when `import.meta.env.DEV`; production stays silent.

## Acceptance criteria

- [ ] On a system with `LANG=zh_CN.UTF-8` (or macOS preferred language Simplified Chinese), launching DashMac for the first time shows all UI in Chinese, including the macOS tray context menu.
- [ ] On an English system, the same first launch shows English everywhere.
- [ ] Settings → Appearance → Language: switching to/from Simplified Chinese updates the main window, the tray panel popover, and the macOS tray context menu — all without restart, all within the same render frame from the user's perspective.
- [ ] Restarting the app preserves the chosen language.
- [ ] Setting `language: 'auto'` after a manual override re-applies the system locale.
- [ ] Byte units (`KB`/`MB`/`GB`), IP addresses, interface names, and percentages remain English/numeric in both languages.
- [ ] `npm test` passes, including the new tests.
- [ ] `npm run build` passes TypeScript strict-mode checks.
